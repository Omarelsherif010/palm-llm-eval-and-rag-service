# Feature Specification: LLM Eval Harness & RAG Service

**Feature Branch**: `001-llm-eval-rag-service`
**Created**: 2026-04-13
**Status**: Draft
**Input**: Palm Outsourcing Senior AI Engineer trial task — two independent deliverables demonstrating prompt reliability evaluation and retrieval-augmented generation skills.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Run Prompt Consistency Evaluation (Priority: P1)

As an AI engineer, I want to run a test suite against an LLM and receive a scored report on response consistency, so that I can identify flaky prompt-answer patterns before they reach production.

The evaluation harness accepts a hand-rolled test set of 8–12 cases. Half of the cases are synthetic (generated by explicit, documented rules such as template substitution, numeric variation, or format permutation). The other half are real-world edge cases drawn from past engineering work (with proprietary details redacted).

The harness MUST include:
- At least one **invariance test**: a set of paraphrased, typo-laden, or noisy variants of the same query where the expected answer MUST remain identical.
- At least one **perturbation test**: a set of reordered or distractor-augmented variants where the answer MAY legitimately change, and the harness measures how much it drifts.

The output is a scored summary using simple, reproducible metrics (exact match, regex pattern match, and/or semantic similarity).

**Why this priority**: This is Question 1 of the trial and the foundational deliverable. It demonstrates core evaluation thinking and can be built and demonstrated without any external service dependency.

**Independent Test**: Run the evaluation script from the command line with no arguments beyond a config or test-set file. It produces machine-readable output (JSON or CSV) plus a human-readable summary. Verify that invariance tests flag when answers diverge and perturbation tests capture drift magnitude.

**Acceptance Scenarios**:

1. **Given** a test set file with 8–12 cases (50% synthetic, 50% real-world), **When** the evaluation script runs against an LLM, **Then** it produces per-case scores and an aggregate consistency report.
2. **Given** an invariance test group with paraphrased queries, **When** the LLM returns different answers for semantically identical queries, **Then** the harness flags the inconsistency and scores it below threshold.
3. **Given** a perturbation test group with reordered/distractor queries, **When** answers drift across variants, **Then** the harness quantifies the drift magnitude rather than simply pass/fail.
4. **Given** the evaluation script, **When** run with `--help` or no valid input, **Then** it prints usage instructions and exits cleanly.

### User Story 2 - Query the RAG Answering Service (Priority: P2)

As a user (or downstream service), I want to send a natural-language query to an HTTP endpoint and receive the top-k most relevant text snippets along with a generated answer grounded in those snippets, so that I can get accurate, traceable answers from a curated corpus.

The service is backed by a small, hand-written text corpus of 10–15 snippets. It exposes a single `POST /answer` endpoint that accepts a query and returns retrieved snippets plus a naive LLM-generated answer. "Naive" means: the retrieved snippets are passed as context in a system message with a simple instruction to answer the query based on the provided context; no chain-of-thought, no multi-turn, no re-ranking — a single LLM call with default temperature.

One guardrail MUST be active on the endpoint (e.g., a denylist of prohibited query topics, a per-user budget/rate cap, or a token-length limiter). The guardrail rationale MUST be documented.

Two index configurations MUST be compared (e.g., cosine similarity vs dot-product, or k=3 vs k=5) with a documented justification for the chosen default.

Two lightweight monitoring metrics MUST be proposed (e.g., request latency percentiles, retrieval hit-rate, embedding drift) with a sketch of how they would be tracked.

**Why this priority**: This is Question 2 of the trial. It builds on different skills (service design, retrieval, guardrails) and is independently deliverable. It depends on no shared code with the eval harness.

**Independent Test**: Start the service locally, send a POST request to `/answer` with a sample query, and verify the response contains retrieved snippets and a generated answer. Trigger the guardrail (e.g., send a denied topic) and verify it is blocked. Compare retrieval results under two configurations.

**Acceptance Scenarios**:

1. **Given** the service is running and the corpus is loaded, **When** a user sends `POST /answer` with a valid query, **Then** the response contains the top-k relevant snippets and a generated answer.
2. **Given** a query that triggers the guardrail (e.g., a denied topic), **When** sent to `POST /answer`, **Then** the request is rejected with a clear explanation and no answer is generated.
3. **Given** two index configurations, **When** the same set of test queries is run against each, **Then** a comparison report shows retrieval quality differences with a documented justification for the default choice.
4. **Given** the service is running, **When** requests are processed, **Then** at least two monitoring metrics (latency and hit-rate or similar) are tracked via structured logs or a metrics endpoint.

### Edge Cases

- What happens when the LLM API is unavailable or rate-limited during evaluation? The harness MUST handle API errors gracefully and report partial results rather than crashing.
- What happens when the query corpus is empty or a snippet file is missing? The RAG service MUST return a clear error, not an empty or hallucinated answer.
- What happens when a query matches no snippets above a relevance threshold? The service MUST indicate low confidence rather than returning irrelevant content.
- What happens when the guardrail check itself fails (e.g., denylist file cannot be loaded)? The service MUST fail closed — reject the request rather than bypassing the guardrail.
- What happens when the eval harness test set contains fewer than 8 or more than 12 cases? The harness MUST warn but still run, rather than refusing.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide an evaluation script that accepts a test set file and produces scored output measuring LLM response consistency.
- **FR-002**: The test set MUST contain 8–12 hand-rolled cases, with approximately 50% synthetic (with documented generation rules) and 50% real-world edge cases (redacted).
- **FR-003**: The test set MUST include at least one invariance test group (paraphrase/typo/noise variants expecting identical answers) and one perturbation test group (order/distractor variants measuring drift).
- **FR-004**: The evaluation script MUST score responses using at least two metrics from: exact match, regex pattern match, semantic similarity.
- **FR-005**: The evaluation script MUST produce both machine-readable output (JSON) and a human-readable summary.
- **FR-006**: System MUST provide an HTTP endpoint `POST /answer` that accepts a natural-language query and returns top-k retrieved snippets plus a generated answer.
- **FR-007**: The retrieval corpus MUST consist of 10–15 hand-written text snippets that are version-controlled with the project.
- **FR-008**: The `POST /answer` endpoint MUST have at least one active guardrail that rejects requests matching prohibited criteria.
- **FR-009**: System MUST compare two index configurations with documented results and a justified default choice.
- **FR-010**: System MUST track at least two monitoring metrics (e.g., latency percentiles, retrieval hit-rate) via structured logs or a dedicated endpoint.
- **FR-011**: All configuration (API keys, model names, thresholds) MUST be externalized to environment variables or configuration files, never hardcoded.
- **FR-012**: Both deliverables MUST be runnable from the command line with a single documented command.

### Key Entities

- **Test Case**: A single evaluation item containing a prompt, expected answer (or answer pattern), test type (invariance/perturbation/standard), and optional metadata (generation rule for synthetic cases, source domain for real-world cases).
- **Test Group**: A collection of related test cases that form an invariance or perturbation set. All cases in an invariance group share the same expected answer.
- **Evaluation Result**: Per-case scores (metric type, score value, pass/fail) and aggregate statistics (mean consistency, failure count, drift magnitude).
- **Corpus Snippet**: A text passage in the RAG knowledge base, with an identifier and optional metadata (topic, source).
- **Query**: A user's natural-language question submitted to the RAG endpoint.
- **Retrieval Result**: The top-k snippets returned for a query, each with a relevance score.
- **Answer Response**: The combined output of retrieved snippets and generated answer, including confidence indicators.
- **Guardrail Rule**: A rule (denylist entry, budget threshold, or filter pattern) that determines whether a request is allowed or blocked.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: The evaluation harness runs end-to-end on the full test set and produces a scored report within 5 minutes (assuming reasonable LLM API latency).
- **SC-002**: Each invariance test group flags when any member answer diverges from the group's expected answer (the group consistency score drops below 1.0 when answers are not identical, with zero false negatives on intentionally divergent responses).
- **SC-003**: The RAG endpoint returns relevant snippets (at least one of the top-k snippets is on-topic) for 80% of test queries drawn from the corpus domain.
- **SC-004**: The guardrail blocks 100% of requests that match prohibited criteria, with zero false negatives.
- **SC-005**: Both deliverables can be set up and run by a reviewer within 10 minutes using only the README instructions.
- **SC-006**: The index configuration comparison report clearly shows which configuration performs better on the test queries, with numeric scores for both.
- **SC-007**: The ship-first-vs-later note identifies concrete, actionable items for immediate and deferred work.

## Assumptions

- The LLM provider API key is available as an environment variable; the system does not manage API key provisioning.
- The evaluation harness and RAG service are independent deliverables that share no runtime code (they may share utility functions but are not coupled).
- The prototype targets local single-developer use; horizontal scaling, authentication, and multi-tenancy are out of scope.
- The text corpus is small enough (10–15 snippets) that in-memory indexing is sufficient; no external vector database is required.
- Reviewers have Python 3.11+ installed and can install dependencies via a requirements file.
- The Loom video and Google Doc are created separately by the engineer and are not generated by the system.
- "Real-world edge cases" are drawn from the engineer's past experience but redacted to remove proprietary information — the specific domain content is flexible.
